Omschrijving

Deze innovatieroute bracht ons de kernprincipes van Domain‑Driven Design (DDD) bij via theorie, quizjes en interactieve oefeningen. Mijn doel: DDD begrijpen vóór het in Full‑stack .NET opdook, zodat ik daar vlotter kon meekomen.

Kern

Kick‑off door Daniël Mertens (Info Support)

Dag 1: Event storming & domein­modellering

We brachten een proces visueel in kaart: events, commands, entities en value objects kregen elk een eigen kleur post‑it. Zo werd het verschil tussen onveranderlijke value objects en identiteits­gebonden entities onmiddellijk duidelijk.

Onze kleur­gecodeerde event‑storming muur

Daarna introduceerden we aggregates: domain commands triggeren domain events die de state wijzigen. Quizjes tussendoor testten onze kennis.

De dag eindigde met het opzetten van een layered/onion architecture skeleton (presentation → application → domain → infrastructure), dit kende we al van Programming Advanced het jaar ervoor.

Dag 2: Application layer & persistence

Application‑services valideren input, halen aggregates op, voeren commands uit en schrijven data terug. Daarna vergeleken we state stored tegen event sourcing; de impact op flexibiliteit en audit‑trail werd duidelijk.

Dankzij de mix van theorie, voorbeelden en hands‑on oefeningen snap ik nu niet alleen wat DDD is, maar kan ik het ook toepassen in echte projecten.

Reflectie

In het begin klonken begrippen als aggregate en value object vaag; nu zie ik hoe een doordacht domein­model de basis vormt voor onderhoudbare software. Het “eerst model → dan code”‑principe neem ik mee naar alle toekomstige projecten.

De route sloot perfect aan bij wat we later in de .NET‑lessen zagen: onion‑architecture en DDD‑concepten kwamen daar terug, maar ik had dankzij deze tweedaagse al een voorsprong. Daarom verdient deze activiteit een plaats in mijn portfolio.
